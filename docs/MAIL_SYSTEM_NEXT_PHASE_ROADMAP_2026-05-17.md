日期：2026-05-17

目标：把当前“可用原型”推进成“可信、可维护、可持续增强的邮件驱动执行系统”。

---

## 1. 总体判断

下一阶段不建议继续按“页面补按钮”推进。  
应该按能力层推进，顺序是：

1. 先稳线程
2. 再稳自动策略
3. 再稳同步与调度
4. 最后做更深的 AI 协商与前端产品化

如果顺序反过来，系统会越来越华丽，也越来越不可信。

---

## 2. 研发阶段总览

建议拆成 4 个阶段。

### Phase 1：核心链路加固

目标：

- 让线程、状态、自动处理行为变得可信

核心结果：

- 线程归并增强
- 自动回信降级为可配置策略
- 显式线程状态机
- 更强的审计与测试基线

### Phase 2：真实邮件处理能力补全

目标：

- 让系统对真实邮箱更稳定、更接近长期使用

核心结果：

- 附件元数据
- 后台轮询/调度
- 更完整的文件夹同步
- 发信失败治理

### Phase 3：AI 协商能力升级

目标：

- 让邮件系统从“会判断”变成“会协商”

核心结果：

- 判断层重构
- 先询问后执行
- 规划邮件结构化抽取
- 联系人级风格与策略

### Phase 4：前端模块化与产品化

目标：

- 让书信台长期可维护，并支撑更复杂的邮件工作流

核心结果：

- `Download.jsx` 拆解
- 更细筛选与视图
- 附件区、联系人轨迹区、协商侧栏

## 3. Phase 1 详细排期

这是最高优先级阶段。

## 3.1 P1-A 线程归并增强

### 目标

从“主题近似归并”升级成“回复链优先归并”。

### 任务

- 在 `mail_messages` 增加字段：
  - `in_reply_to`
  - `references_json`
- 在 IMAP 解析阶段提取：
  - `In-Reply-To`
  - `References`
- 重写 `_find_existing_thread_id()` 的匹配优先级：
  - 先按 `In-Reply-To` 命中已有消息
  - 再按 `References` 反查已有消息
  - 再按 `internet_message_id`
  - 最后才回退到 `subject_normalized`
- 增加“线程误合并防护”：
  - 同主题但参与人差异过大时不自动并入

### 交付标准

- 同一真实回复链能稳定回到原线程
- 同主题不同事务的误合并明显减少
- 新增回归测试覆盖 6 个以上场景

### 涉及文件

- [mail_service.py](/data/sda/tanzheng/Desktop/My_Claw/local-gateway/services/mail_service.py)
- [test_mail_service.py](/data/sda/tanzheng/Desktop/My_Claw/local-gateway/test/test_mail_service.py)

## 3.2 P1-B 线程状态机显式化

### 目标

把“客观状态”和“主观判断”拆开。

### 任务

- 设计线程状态字段分层：
  - 客观状态
    - `has_new_inbound`
    - `has_pending_draft`
    - `is_archived`
    - `last_actor`
  - 评估状态
    - `reply_level`
    - `decision_status`
    - `risk_level`
    - `mail_kind`
- 将 `_refresh_thread_state()` 改成两段：
  - 事实汇总
  - 评估推断
- 把 `needs_reply` 的逻辑从简单布尔改成更可解释的派生结果

### 交付标准

- 状态字段语义不再混乱
- 前端队列和 dashboard 的统计口径更稳定
- 能明确区分“对方最后发言”和“必须回复”不是一回事

## 3.3 P1-C 自动处理策略分层

### 目标

把当前激进的自动回信链路收束成可配置策略。

### 任务

- 新增自动策略枚举：
  - `draft_only`
  - `draft_and_notify`
  - `auto_send`
- 默认策略改为 `draft_and_notify`
- `auto_handle_incoming_mail()` 重构：
  - 先判断邮件类型
  - 先生成草稿
  - 先进入待决策池
  - 只有在明确低风险、明确授权场景才允许自动发送
- `mail_agent_runs` 增加更细结果类型：
  - `skipped_non_direct`
  - `draft_created`
  - `user_confirmation_required`
  - `sent`
  - `failed`

### 交付标准

- 默认不会再对普通用户来信直接发正式回复
- 高风险规划邮件统一走“草稿 + 询问”路径
- 自动行为可以审计

## 3.4 P1-D 测试与审计基线加强

### 目标

让核心链路具备回归保护。

### 任务

- 为以下场景补测试：
  - 线程归并
  - 多轮往返信件后 `needs_reply` 变化
  - 自动策略三档行为
  - portal token 校验
  - 发信失败后的草稿状态
- 新增简单的 agent audit 查询接口

### 交付标准

- 邮件核心领域测试显著增加
- 每次改判断逻辑不再完全靠手工点界面验证

---

## 4. Phase 2 详细排期

## 4.1 P2-A 附件域模型

### 目标

让真实邮件中的附件不再是黑洞。

### 任务

- 新增 `mail_attachments` 表
- 同步时提取：
  - 文件名
  - MIME type
  - 大小
  - Content-ID
  - 是否 inline
- 第一阶段不下载大附件本体，只做元数据和按需提取接口

### 交付标准

- 线程详情里能看到附件列表
- AI/任务生成时可感知“这封信带附件”

## 4.2 P2-B 后台轮询调度器

### 目标

减少“必须手点同步”的依赖。

### 任务

- 新增轮询任务：
  - 每 2 到 5 分钟检查启用同步的账户
- 支持按账户开关
- 支持失败冷却时间
- 记录最近一次自动同步时间与状态

### 交付标准

- 新邮件无需手工点同步也能入库
- 重复失败时不会疯狂打 IMAP

## 4.3 P2-C 发信失败治理

### 目标

把发信失败从“一个错误字符串”升级成“可处理状态”。

### 任务

- 给 `mail_drafts.status` 扩展更细状态：
  - `draft`
  - `queued`
  - `sending`
  - `sent`
  - `failed`
  - `retry_waiting`
- 记录失败原因分类：
  - SMTP 认证失败
  - 网络超时
  - 收件人无效
  - 服务端拒绝
- 提供手动重试接口

### 交付标准

- 用户能知道“为什么没发出去”
- 系统可对临时失败做再次尝试

## 4.4 P2-D portal 安全增强

### 目标

让移动处理页不只是能用，还更稳。

### 任务

- token 增加过期时间
- token 与线程、动作范围绑定
- 支持失效与刷新
- 为 portal 动作增加更明确的错误页与来源记录

### 交付标准

- portal 链接长期暴露的风险降低
- 动作权限边界更清晰

---

## 5. Phase 3 详细排期

## 5.1 P3-A 判断层重构

### 目标

从“关键词命中”升级为“规则 + LLM 评估”的双层判断。

### 任务

- 保留规则层作为快速、可解释的第一层
- 增加可选 LLM 评估层：
  - 是否必须回复
  - 是否属于规划邮件
  - 风险等级
  - 建议动作
- 将 LLM 输出结构化写回线程

### 交付标准

- 复杂邮件判断质量提升
- 前端理由字段更像真实参谋判断，而不是模板句

## 5.2 P3-B 先询问后执行的协商流

### 目标

让系统更像参谋，而不是越权代办者。

### 任务

- 对规划邮件默认生成“询问卡”
- 询问卡动作结构化：
  - 起草回复
  - 转任务
  - 转日程候选
  - 稍后再问
  - 直接归档
- 用户动作写回线程状态

### 交付标准

- 高风险邮件不会直接被自动处理
- 邮件协商进入系统主工作流

## 5.3 P3-C 联系人级策略

### 目标

让系统开始记住“这是谁、我通常怎样回应他”。

### 任务

- 联系人表或联系人策略表
- 记录：
  - 常用称呼
  - 默认语气
  - 自动化允许级别
  - 重要联系人标记
- 草稿生成时引用联系人策略

### 交付标准

- 回信语气更稳定
- 自动策略更细，不再一刀切

## 5.4 P3-D 规划邮件结构化抽取

### 目标

让邮件中的安排真正变成计划对象，而不是一句摘要。

### 任务

- 提取：
  - 日期
  - 时间段
  - 截止时间
  - 地点
  - 所需动作
- 生成候选任务或候选日程块

### 交付标准

- 与面试、会议、体检、付款、材料提交相关的邮件能直接进入计划层

## 6. Phase 4 详细排期

## 6.1 P4-A 前端模块化拆分

### 目标

让书信台从“大组件”变成“可长期维护的页面系统”。

### 任务

- 从 `Download.jsx` 拆出：
  - `MailDashboardSummary`
  - `DecisionQueueSection`
  - `MailRail`
  - `OpenLetterPanel`
  - `MailComposerModal`
  - `mail-rendering.tsx`
- 将链接渲染、HTML 清洗逻辑独立

### 交付标准

- 页面逻辑明显减重
- 后续迭代附件区、联系人区时不会继续堆到单文件

## 6.2 P4-B 更丰富的书信台视图

### 目标

增强筛选、搜索和工作流效率。

### 任务

- 增加筛选：
  - 联系人
  - 风险等级
  - 是否有草稿
  - 是否已转任务
  - 时间范围
- 增加 archive 时间浏览
- 增加线程搜索结果高亮

### 交付标准

- 书信台从“单线程处理页”进化为“邮件工作台”

## 6.3 P4-C 协商侧栏

### 目标

把 `AiChat` 的 `mail_consult` 从 prompt 注入升级成正式界面。

### 任务

- 在右侧加入协商区或抽屉
- 固定显示：
  - 线程摘要
  - AI 判断
  - 建议回复策略
  - 建议任务/日程动作
- 支持一键把协商结果写回草稿或任务

### 交付标准

- 邮件与 AI 协商不再跳出主工作台

## 6.4 P4-D 正文与附件阅读体验完善

### 目标

让真实邮件阅读更稳。

### 任务

- 图片与远程资源策略
- 转发内容折叠
- 更复杂表格邮件适配
- 附件卡片区
- 联系人轨迹区

### 交付标准

- 书信台可承载更复杂的真实邮件样式

## 7. 推荐执行顺序

建议严格按下面顺序推进：

1. `P1-A 线程归并增强`
2. `P1-B 线程状态机显式化`
3. `P1-C 自动处理策略分层`
4. `P1-D 测试与审计基线加强`
5. `P2-B 后台轮询调度器`
6. `P2-C 发信失败治理`
7. `P2-A 附件域模型`
8. `P2-D portal 安全增强`
9. `P3-B 先询问后执行的协商流`
10. `P3-A 判断层重构`
11. `P3-D 规划邮件结构化抽取`
12. `P3-C 联系人级策略`
13. `P4-A 前端模块化拆分`
14. `P4-B/P4-C/P4-D` 并行推进

## 8. 我建议你现在就立项的第一批任务

如果你要开始实作，我建议直接把第一批任务定成这 6 条：

### Task 1

为 `mail_messages` 增加 `in_reply_to` 与 `references_json`，完成 schema migration。

### Task 2

重写 `_parse_imap_message()` 与 `_find_existing_thread_id()`，上线更稳定的线程归并。

### Task 3

为线程设计显式状态机字段，并重构 `_refresh_thread_state()`。

### Task 4

把自动回信改成策略分层，默认切换到 `draft_and_notify`。

### Task 5

补齐邮件领域核心测试，特别是线程、自动策略和发信失败场景。

### Task 6

给书信台增加 agent/audit 可见区，至少能看到自动处理是否发生、做了什么、为何跳过。

## 9. 里程碑定义

### 里程碑 M1

线程稳定，自动策略不越权，核心测试补齐。  
达到这个里程碑后，系统才算“可信原型”。

### 里程碑 M2

同步和发信长期运行更稳定，portal 更安全，附件开始进入系统。  
达到这个里程碑后，系统才算“真实可日用”。

### 里程碑 M3

AI 判断、协商、规划抽取正式成型。  
达到这个里程碑后，系统才开始像真正的书信参谋。

### 里程碑 M4

前端模块化完成，书信台可以稳定承载复杂流程。  
达到这个里程碑后，系统才适合继续做更深的产品化体验。

---

## 10. 结论

下一阶段最重要的不是继续往界面上撒更多功能，而是先把这封正在成形的系统之信写稳、写准、写得值得托付。

先让每一封来信真正回到正确线程，  
再让每一次 AI 介入都克制而有分寸，  
再让同步、草稿、发送、协商这些动作像钟表里的齿轮一样咬合。

当这些基础稳住之后，书信台才配得上继续长出更复杂的浪漫与更深的执行力。
