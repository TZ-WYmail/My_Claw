# 前端功能实现基线与模块化开发台账

日期：2026-05-16  
适用范围：`local-gateway/frontend` + `local-gateway/routers`  
目标：把“前端已经呈现出来、但后端能力还不完整”的部分系统化梳理，形成后续模块化开发的执行基线。

---

## 0. 结论先行

当前产品的问题不是“页面不够多”，而是“页面已经承诺了很多能力，但后端与状态模型还没有把这些能力真正兜住”。

最明显的缺口有 7 类：

- 番茄钟只具备基础开始 / 完成 / 中断，不具备完整会话生命周期。
- 任务、日历、番茄钟还没有统一成一个真正的时间调度系统。
- `AiChat` 已经能预览和确认规划，但规划结果的回写、追踪、失败恢复还不够完整。
- 工作流页面已有 CRUD 和执行入口，但离“可观测、可调试、可复用”的自动化系统还有距离。
- 同步和下载页面已有入口，但还偏演示型接口，缺少冲突、重试、审计和异常分层。
- 习惯和笔记页面已经能操作数据，但没有形成长期追踪和关联网络。
- 设置页只覆盖聊天与通知，没有沉淀成应用级能力总线。

因此后续开发不应再按“页面补小功能”推进，而应按平台模块推进：

1. 时间与专注系统
2. 任务与排程系统
3. AI 规划落地系统
4. 工作流执行系统
5. 同步与下载基础设施
6. 笔记 / 习惯 / 通知联动系统

---

## 1. 方法

本台账按每个页面拆成 6 个维度：

- 当前前端入口
- 完整目标功能
- 推荐实现形式
- 当前后端支撑
- 缺失能力
- 模块化开发拆分

说明：

- `AiChat` 作为“参谋手稿台”保留当前主结构，不做大改，只补强功能闭环。
- 页面“呈现形式”沿用现有“行动地图册 / 纸面作战室”世界观，但重点是功能承载方式，不是单独做一版视觉提案。

## 3. 分页面 Brainstorm 与实现台账

## 3.1 Dashboard / Today

### 当前前端入口

- 拉取 `/api/dashboard`
- 拉取 `/api/streak`
- 拉取 `/api/task` 的 `get_pending_tasks`
- 拉取 `/api/advanced/pomodoro/status`
- 拉取任务子任务详情

### 完整目标功能

- 今日主线任务总览
- 今日时间块与风险提示
- 当前专注会话状态
- 今日任务推进节奏
- 今天必须完成 / 可以延后 / 已卡住任务分层
- AI 给出的今日推进建议
- 从首页直接进入开始专注、补子任务、记笔记、挪期、开 AI 规划

### 推荐实现形式

定位为“今日行动总览双页”：

- 左页：今天的主线、节奏、当前专注
- 右页：风险雷达、阻塞任务、快速决策
- 番茄钟不是一个孤立卡片，而是“今日推进节奏器”
- 待办列表不做纯列表，改成“今日战线”分区：
  - 正在推进
  - 下一步
  - 等待处理
  - 已逾期

### 当前后端支撑

- 有基础 dashboard 汇总数据
- 有 streak
- 有 pending tasks
- 有番茄钟状态
- 有子任务查询

### 缺失能力

- `get_pending_tasks` 不能稳定表达“今日任务”
- 缺少今日优先级重新排序逻辑
- 缺少“阻塞原因 / 下一动作”字段
- 缺少今日剩余专注时段估算
- 缺少今日任务与日历事件合并视图
- 缺少首页级番茄钟历史与完成率摘要

### 模块化开发拆分

- `dashboard_today_summary`
- `dashboard_focus_widget`
- `dashboard_risk_panel`
- `dashboard_quick_actions`
- 依赖平台模块：
  - `timebox_engine`
  - `pomodoro_session`
  - `task_schedule_projection`

### 优先级

高。它依赖任务、日历、番茄钟基础模块打通后再抬升。

## 3.2 Tasks

### 当前前端入口

- 任务 CRUD
- 批量完成 / 删除
- 全量任务筛选
- 周计划
- 子任务 CRUD
- 番茄钟开始 / 完成 / 中断
- 关联笔记

### 完整目标功能

- 任务完整生命周期：
  - 收集
  - 澄清
  - 排期
  - 执行
  - 复盘
- 支持任务与项目、上下文、能量、预计时长、截止时间、重复规则关联
- 周计划和日计划联动
- 子任务、阻塞项、下一动作、参考资料完整挂接
- 专注记录自动写回任务
- 完成任务后自动产生日志或复盘条目

### 推荐实现形式

定位为“周排程折页 + 任务索引柜”：

- 上半区是本周排程折页
- 下半区是任务抽屉：
  - 待澄清
  - 已排程
  - 执行中
  - 阻塞
  - 已完成
- 任务详情页是“档案卡”而不是普通弹窗：
  - 元信息
  - 子任务
  - 关联笔记
  - 专注记录
  - 变更历史

### 当前后端支撑

- 基础任务 CRUD
- batch 操作
- 周计划接口
- 子任务接口
- 任务详情接口
- 番茄钟基础接口

### 缺失能力

- 没有真正统一的任务状态机
- 缺少任务历史、操作审计、字段级修改记录
- 缺少任务依赖关系
- 缺少重复任务规则
- 缺少任务排序策略持久化
- 缺少任务与日历事件的双向绑定
- 番茄钟结束后没有稳定写回任务统计
- 缺少“下一动作”与“阻塞原因”作为一等字段

### 模块化开发拆分

- `task_domain_model`
- `task_list_and_filters`
- `task_detail_record`
- `subtask_manager`
- `task_note_linker`
- `task_focus_integration`
- `weekly_planning_board`

### 优先级

最高。任务页是后续日历、番茄钟、AI 规划回写的核心枢纽。

## 3.3 Calendar

### 当前前端入口

- 月视图
- 日事件列表
- 事件增删
- 番茄钟状态
- 从任务启动专注

### 完整目标功能

- 月 / 周 / 日三级日历
- 任务排程、手动事件、专注时段统一落点
- 支持拖拽调整时间
- 支持冲突检测
- 支持重复事件
- 支持从日历直接创建任务、笔记、专注块
- 支持“今日空档推荐”

### 推荐实现形式

定位为“月度地形图 + 日程坐标页”：

- 月视图用于观察负载分布与密度
- 选中单日后打开“当日坐标页”
- 每个时间块可承载：
  - 任务
  - 事件
  - 专注块
  - 提醒
- 与任务系统共用同一时间模型，不再各自维护

### 当前后端支撑

- 月视图接口
- 日事件查询
- 事件新增 / 删除
- 番茄钟状态

### 缺失能力

- 缺少周 / 日视图接口
- 缺少事件编辑接口的一致能力
- 缺少重复规则
- 缺少拖拽排程语义
- 缺少任务事件绑定模型
- 缺少冲突检测
- 缺少专注块自动落盘逻辑

### 模块化开发拆分

- `calendar_domain_model`
- `calendar_month_board`
- `calendar_day_timeline`
- `schedule_drag_and_commit`
- `task_event_binding`
- `focus_block_projection`

### 优先级

高。应与任务和番茄钟一起作为同一阶段实现。

## 3.4 Notes

### 当前前端入口

- 笔记列表 / 搜索
- 笔记 CRUD
- 可关联任务

### 完整目标功能

- 日常记录、任务笔记、项目资料、AI 产出沉淀统一收纳
- 笔记支持标签、来源、状态、关联对象
- 支持从任务、AI 规划、工作流执行记录反链到笔记
- 支持长文编辑与结构化摘录
- 支持最近编辑、固定、归档、引用关系

### 推荐实现形式

定位为“线索剪报页 + 档案抽屉”：

- 左侧是笔记抽屉和过滤器
- 右侧是当前笔记剪报页
- 笔记不是孤立富文本，而是“可挂接到任务 / AI / 工作流”的知识节点

### 当前后端支撑

- 基础 CRUD
- 列表与搜索

### 缺失能力

- 缺少标签系统
- 缺少归档 / 固定
- 缺少版本历史
- 缺少引用关系与反向链接
- 缺少附件支持
- 缺少结构化模板
- 缺少 AI 规划结果落地为笔记的通道

### 模块化开发拆分

- `note_domain_model`
- `note_editor`
- `note_index_and_filters`
- `note_link_graph`
- `task_note_reference`
- `ai_note_capture`

### 优先级

中高。它不是基础设施，但会被任务和 AI 明显依赖。

## 3.5 Habits

### 当前前端入口

- 习惯列表
- 创建习惯
- 打卡
- 删除习惯

### 完整目标功能

- 支持习惯目标、频率、单位、提醒、统计周期
- 支持打卡历史、连续记录、补打卡
- 支持习惯与任务、日历、番茄钟协同
- 支持趋势分析与异常提醒
- 支持“恢复计划”，识别中断后的回归路径

### 推荐实现形式

定位为“训练轨迹板”：

- 每个习惯是一条训练轨迹，不只是一个数字卡
- 核心表现不是“总次数”，而是：
  - 连续性
  - 周期完成率
  - 最近波动
  - 今日是否需要执行

### 当前后端支撑

- 习惯 CRUD
- checkin
- stats

### 缺失能力

- 缺少打卡历史明细接口
- 缺少补打卡 / 撤销打卡
- 缺少周期规则表达
- 缺少提醒机制
- 缺少与番茄钟或任务联动
- 缺少异常检测与趋势分析字段

### 模块化开发拆分

- `habit_domain_model`
- `habit_schedule_rules`
- `habit_checkin_history`
- `habit_analytics`
- `habit_reminder_bridge`

### 优先级

中。可在任务 / 日历 / 番茄钟稳定后展开。

## 3.6 AiChat

### 当前前端入口

- 聊天配置
- SSE 流式对话
- 会话历史
- 新建 / 删除会话
- AI 规划预览 / 确认 / 重规划 / 接受重规划
- 载入今日待办
- 系统时间显示

### 完整目标功能

- 顾问对话、任务编成、计划确认、回写落地形成闭环
- AI 不是只输出文本，而是能产出：
  - 任务草案
  - 排程建议
  - 风险提示
  - 复盘提纲
  - 笔记摘要
- 每次规划都能追踪：
  - 来源会话
  - 生成时间
  - 影响的任务
  - 是否确认
  - 是否被后续重规划覆盖
- 支持“今天怎么推进”“本周怎么排”“这个任务怎么拆”三种主场景

### 推荐实现形式

保留“参谋手稿台”主结构，不做大改，只补强功能分层：

- 左页：回合对话、反思、规划者视角
- 右页：当前战报、规划预览、待确认变更
- AI 的关键价值不是更炫的聊天界面，而是“计划对象化”
- 所有规划结果必须能映射到明确的数据对象，而不是停留在 Markdown

### 当前后端支撑

- 聊天、流式输出、配置、测试
- 会话历史
- AI 规划 preview / confirm / replan / accept
- 系统时间接口

### 缺失能力

- 缺少规划版本链和审计记录
- 缺少确认失败后的补偿和回滚可视化
- 缺少规划结果与任务 / 日历 / 笔记的统一回写总线
- 缺少“仅看今日 / 本周 / 某项目”上下文载入协议
- 缺少 AI 对已有任务真实引用的置信度表达
- 缺少计划执行后偏差反馈，再回喂给 AI

### 模块化开发拆分

- `ai_chat_session`
- `ai_context_loader`
- `ai_plan_object_model`
- `ai_plan_confirmation_pipeline`
- `ai_plan_revision_audit`
- `ai_task_calendar_note_bridge`

### 优先级

高，但依赖任务 / 日历 / 笔记对象模型先稳定。  
注意：这里是“功能补强”，不是视觉重做。

## 3.7 Workflows

### 当前前端入口

- 工作流列表
- 创建
- 启停
- 执行
- 删除
- 查看执行记录

### 完整目标功能

- 工作流定义、验证、试跑、执行、审计完整闭环
- 支持触发器、步骤、条件分支、重试、超时和失败策略
- 支持模板化复用
- 支持查看单次执行的详细步骤日志
- 支持工作流与任务 / 下载 / 通知联动

### 推荐实现形式

定位为“自动化装配册”：

- 列表页展示“装配卡”
- 详情页展示“步骤链”
- 执行记录不是一行列表，而是可展开的执行卷宗：
  - 输入
  - 每一步结果
  - 失败点
  - 输出

### 当前后端支撑

- list / create / delete / toggle / execute
- execution history
- workflow types

### 缺失能力

- 缺少步骤级调试信息
- 缺少 schema 校验反馈
- 缺少模板系统
- 缺少触发器测试入口
- 缺少失败重试与补偿策略
- 缺少运行日志下载 / 查看
- 缺少工作流结果和业务对象的关联

### 模块化开发拆分

- `workflow_definition_model`
- `workflow_builder`
- `workflow_execution_audit`
- `workflow_trigger_lab`
- `workflow_templates`
- `workflow_business_bridge`

### 优先级

中高。适合在基础任务系统稳定后单独推进。

## 3.8 Sync

### 当前前端入口

- 状态
- 设备
- offline queue
- push / pull / full / offline sync

### 完整目标功能

- 多设备状态总览
- 本地待同步变更队列
- 同步结果回执
- 冲突检测与人工处理
- 失败重试与重放
- 最近同步对象审计

### 推荐实现形式

定位为“回传收发台”：

- 左侧看设备和链路状态
- 中间看待回传对象

### 当前后端支撑

- 基础状态
- 设备列表
- queue
- push / pull / full / offline sync

### 缺失能力

- 缺少同步结果明细
- 缺少每条变更的重试状态
- 缺少设备能力 / 版本 / 最后成功同步语义
- 缺少失败分类和用户可操作建议

### 模块化开发拆分

- `sync_domain_model`
- `sync_device_registry`
- `sync_change_queue`
- `sync_conflict_resolution`
- `sync_execution_receipts`

中高。属于基础设施强化项，应在业务对象模型稳定后推进。

## 3.9 Download

### 当前前端入口

- 队列拉取
- 带宽限制
- 新增下载
- 暂停 / 恢复 / 取消

### 完整目标功能

- 下载任务完整状态机
- 来源验证
- 下载进度 / 速率 / 剩余时间
- 自动重试
- 下载完成回调
- 与任务、工作流、通知联动
- 历史记录与失败诊断

### 推荐实现形式

定位为“港口调度图 + 货箱转运架”：

- 任务列表按状态进入不同“货位”
- 每个下载任务显示来源、大小、完整性、重试次数、目标位置
- 带宽不是单独设置项，而是调度器总阀门

### 当前后端支撑

- queue
- add to queue
- pause / resume / cancel
- bandwidth

### 缺失能力

- 缺少下载历史
- 缺少失败原因分类
- 缺少校验和 / 文件完整性状态
- 缺少自动重试策略
- 缺少批量操作
- 缺少目标目录与后处理动作
- 缺少通知回执

### 模块化开发拆分

- `download_domain_model`
- `download_queue_scheduler`
- `download_integrity_and_retry`
- `download_history`
- `download_notification_bridge`

### 优先级

中。适合和同步基础设施一起做。

## 3.10 Sandbox

### 当前前端入口

- 代码输入
- 语言与超时参数
- 执行结果

### 完整目标功能

- 多语言执行
- 历史记录
- 模板片段
- 运行参数可复用
- 标准输出 / 标准错误 / 资源用量分离展示
- 支持从工作流或 AI 结果直接送入试验

### 推荐实现形式

定位为“试验托盘”：

- 输入区、运行参数、结果区像桌面器具
- 结果不是一块大文本，而是：
  - 输出
  - 错误
  - 用时
  - 退出码

### 当前后端支撑

- 基础执行接口

### 缺失能力

- 缺少执行历史
- 缺少模板与片段
- 缺少资源指标
- 缺少结果持久化
- 缺少与 AI / 工作流互通入口

### 模块化开发拆分

- `sandbox_run_record`

## 5. 分阶段模块化开发建议

## Phase 1：时间 / 任务 / 番茄钟基础层

目标：先修“前端最常用、但后端语义不完整”的主干。

包含模块：

- `system_time_core`
- `task_domain_model`
- `task_schedule_core`
- `pomodoro_session`
- `calendar_day_timeline`

验收标准：

- 今日任务语义统一
- 番茄钟可完整闭环并写回任务
- 任务与日历能共享时间块

## Phase 2：AI 规划落地层

目标：把 `AiChat` 从“会聊天”变成“能落地编成计划”。

包含模块：

- `ai_context_loader`
- `ai_plan_core`
- `ai_plan_confirmation_pipeline`
- `ai_task_calendar_note_bridge`

验收标准：

- 每份 AI 规划都有版本和回执
- 规划确认后能稳定生成任务 / 时间块 / 笔记引用

## Phase 3：知识与行为层

目标：让笔记和习惯不再只是独立页面。

包含模块：

- `note_domain_model`
- `note_link_graph`
- `habit_domain_model`
- `habit_checkin_history`
- `notification_runtime`

验收标准：

- 任务、AI、习惯、笔记之间可互相挂接
- 习惯支持趋势与提醒

## Phase 4：自动化与基础设施层

目标：补齐执行、同步、下载这些“已出现在前台、但后台未完全工程化”的能力。

包含模块：

- `workflow_runtime`
- `workflow_builder`
- `sync_core`
- `download_runtime`

验收标准：

- 工作流执行可调试
- 同步可看冲突和回执
- 下载可看历史、重试和完整性

## Phase 5：设置与系统治理层

目标：把配置从零散表单升级为统一系统治理面板。

包含模块：

- `settings_schema_registry`
- `settings_connection_tests`
- `settings_history_and_rollback`

验收标准：

- 配置可校验、可测试、可回滚
- 各模块配置不再分散定义

## 6. 立即执行建议

如果下一步开始进入代码开发，建议严格按下面顺序推进：

1. 先做 `system_time_core + pomodoro_session`，因为它已经影响 Dashboard、Tasks、Calendar、AiChat。
2. 再做 `task_domain_model + task_schedule_core`，把任务和排程语义收拢。
3. 然后做 `ai_plan_core`，避免 AI 继续输出“无法稳定回写”的半结构结果。
4. 再进入 `note / habit` 与 `workflow / sync / download` 两条并行线。

不建议的推进方式：

- 不要先做单页视觉重写。
- 不要继续在每个页面里各自维护一套本地状态映射。
- 不要继续把“是否能显示出来”当成“功能已经实现”。

## 7. 当前最需要警惕的技术债

- 页面已经有复杂交互，但后端仍以轻量接口拼接，语义层不稳。
- 多个页面对相同对象有各自的本地标准化逻辑，容易继续分叉。
- 番茄钟、同步、下载都已有前端入口，用户会自然认为它们是完整系统，因此不能再按“演示能力”维护。
- `AiChat` 如果没有统一计划对象模型，后续每加一个动作都会继续堆条件分支。

## 8. 本台账对应的下一步开发起点

建议下一轮直接进入下面这个开发主题：

`Phase 1A：system_time_core + pomodoro_session + task_schedule_core`

原因：

- 用户已经明确指出番茄钟类能力前后端不闭环。
- 这三块同时是 Dashboard、Tasks、Calendar、AiChat 的共同基础。
- 先把这层打稳，后面的 AI 规划、习惯提醒、日历排程才不会反复返工。
