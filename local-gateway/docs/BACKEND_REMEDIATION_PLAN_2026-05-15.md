# Backend Remediation Plan — 2026-05-15

## 1. 背景

当前仓库前端主线已基本成型，但后端仍存在“主流程可用、关键能力不完整”的问题。主要表现为：

1. 有接口名，但无真实业务实现
2. 多页面依赖前端拼装多个接口，后端缺少聚合层
3. 提醒、工作流等能力存在“尽力而为”的静默失败路径
4. 批量能力、详情聚合能力不足，导致前端交互成本偏高

本文件用于约束后续后端修复顺序，避免继续零散修补。

---

## 2. 当前确认的后端缺口

### 2.1 P0 / 明确未完成

#### A. 批量任务更新未实现
- 文件：`local-gateway/routers/advanced_features.py`
- 现状：`POST /api/advanced/tasks/batch-update` 仅返回伪造 `updated`
- 影响：
  - 无法支持真实批量改期
  - 无法支持批量优先级调整
  - 前端无法安全调用

#### B. 任务详情缺少聚合接口
- 现状：
  - 任务基础信息、关联笔记、子任务、番茄钟状态、本周上下文需要前端拆开请求
- 影响：
  - 前端详情抽屉请求碎片多
  - 状态同步复杂
  - 后续移动端/其他客户端难复用

### 2.2 P1 / 可用但不稳

#### C. 通知提醒失败不可观测
- 文件：`local-gateway/services/task_service.py`
- 文件：`local-gateway/services/notification_service.py`
- 现状：
  - 调度失败直接 `pass`
  - 调用方无法知道提醒是否注册成功
- 影响：
  - 任务看似创建成功，但提醒可能根本未生效

#### D. 工作流调度只有简化 cron 支持
- 文件：`local-gateway/services/workflow_service.py`
- 现状：
  - 仅支持极少数模式判断
- 影响：
  - 配置能力和用户预期不一致

---

## 3. 修复目标

本轮后端修复只解决“主线不完整”问题，不扩散到所有边缘模块。

### 目标 1：让批量任务更新可真实使用
- 支持批量更新：
  - `priority`
  - `due_time`
  - `tags_add`
  - `tags_remove`
- 返回每个任务的真实处理结果

### 目标 2：提供任务详情聚合接口
- 新增任务详情接口，至少聚合：
  - task
  - notes
  - subtasks
  - active_pomodoro
  - weekly_neighbors

### 目标 3：补关键测试
- 服务层测试
- API 层定向测试

---

## 4. 执行阶段

## Phase B1：批量任务更新补齐

### 任务
1. 在服务层增加批量更新逻辑
2. 路由层接真实处理结果
3. 支持标签增删
4. 补单元测试

### 验收
- 调用 `/api/advanced/tasks/batch-update` 后数据库真实变化
- 返回结果可区分成功/失败任务

---

## Phase B2：任务详情聚合接口

### 任务
1. 新增任务详情服务函数
2. 新增任务详情 API
3. 聚合任务、笔记、子任务、番茄钟、本周上下文
4. 补测试

### 验收
- 前端只需 1 次主请求即可拿到详情主体数据
- 返回结构稳定，可供后续页面复用

---

## Phase B3：提醒链路可观测性

### 任务
1. 任务创建/更新时返回提醒注册 warning
2. 记录调度失败原因
3. 补最小测试

### 验收
- 提醒失败时不再静默吞掉全部信息

---

## 5. 本次准备立即执行的范围

本轮先执行：

1. **Phase B1：批量任务更新补齐**
2. **Phase B2：任务详情聚合接口**
3. 为上述两项补测试

暂不执行：
- 完整 cron 解析
- 大规模通知系统重构
- 同步系统重构

---

## 6. 风险控制

1. 不改现有任务主表结构
2. 不回滚用户已有未提交改动
3. 优先新增函数和接口，避免大面积侵入旧逻辑
4. 每阶段后跑定向测试

---

## 7. 完成标准

满足以下条件才算后端主线修复完成：

1. 批量任务更新真实可用
2. 任务详情聚合接口可用
3. 对应测试通过
4. 前端不再依赖过多碎片请求才能打开任务详情
