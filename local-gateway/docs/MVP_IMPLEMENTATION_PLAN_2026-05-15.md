# LocalCommandCenter MVP 实施计划

> 文档版本: 1.0
> 更新日期: 2026-05-15
> 依据文档: `MVP_DEV_BREAKDOWN_2026-05-15.md`、`MVP_API_CONTRACT_2026-05-15.md`、`MVP_DB_SCHEMA_2026-05-15.md`

---

## 一、目标

本实施计划用于把 MVP 文档体系转化为实际开发顺序，控制范围，减少返工。

原则是：

1. 先打通主链路
2. 再补联动
3. 最后统一视觉和状态

---

## 二、实施顺序总览

建议按照以下顺序推进：

### Phase 1：基础收敛

- 收口导航
- 确定 MVP 页面
- 数据模型迁移
- 建立 MVP API 适配层

### Phase 2：核心页面实现

- 今天工作台
- 任务页
- 日历页
- 笔记页

### Phase 3：AI 辅助与联动

- AI 助手页
- 任务/日历/笔记联动

### Phase 4：视觉、状态与验收

- UI 统一
- 空状态/错误态补齐
- 主路径验收

---

## 三、详细阶段计划

## Phase 1：基础收敛

### 目标

让工程结构先适配 MVP，而不是在旧结构上硬堆页面。

### 任务

#### 1. 导航收口

- 重构前端导航
- 隐藏下载、工作流、同步、沙盒的主入口
- 统一页面命名

#### 2. 建立 MVP API 层

- 新增 `/api/mvp/*` 路由层，或建立稳定 adapter
- 禁止新页面直接耦合旧杂糅接口

#### 3. 数据库迁移

- 新增 `projects`
- 新增 `subtasks`
- 新增 `task_note_links`
- 新增 `note_templates`
- 新增 AI 相关 3 张表
- 补充 `tasks` 新字段

#### 4. 服务层适配

- 首页聚合服务
- 任务今日/周视图服务
- 日历周/月视图服务
- 笔记最近列表服务

### 输出物

- 新导航骨架
- MVP API 框架
- 数据迁移脚本
- 服务层骨架

### 完成标志

- MVP 页面可挂接到稳定接口
- 数据字段足够支持任务/日历/笔记联动

---

## Phase 2：核心页面实现

### 目标

打通“首页 -> 任务 -> 日历 -> 笔记”主链路。

### 任务

#### 1. 今天工作台实现

- 今日重点任务卡
- 今日日程摘要卡
- 逾期提醒卡
- 最近笔记卡
- 快速创建入口

#### 2. 任务页实现

- 今日视图
- 周视图
- 创建/编辑任务弹层
- 子任务展示
- 项目字段

#### 3. 日历页实现

- 周视图主模式
- 月视图概览
- 冲突提示
- 任务时间调整

#### 4. 笔记页实现

- 列表 + 编辑 + 预览
- 模板入口
- 任务关联展示

### 输出物

- 可用的四个核心页面
- 页面间跳转链路

### 完成标志

- 用户可独立完成“创建任务 -> 安排时间 -> 创建关联笔记”

---

## Phase 3：AI 辅助与联动

### 目标

让 AI 成为主链路辅助层，而不是独立系统。

### 任务

#### 1. AI 助手页实现

- 快捷任务入口
- 任务型 AI 能力
- 对话区
- 建议展示区

#### 2. AI 建议确认机制

- 展示理解结果
- 展示建议动作
- 用户确认后应用

#### 3. 联动实现

- 任务时间调整同步日历
- 任务创建笔记
- 笔记回看任务
- AI 回填任务和笔记

### 输出物

- AI 助手 MVP
- 主链路联动完成

### 完成标志

- AI 能帮助用户完成任务安排、拆解、整理，而不会越界执行

---

## Phase 4：视觉、状态与验收

### 目标

让 MVP 不是只有流程能跑，而是真正可交付。

### 任务

#### 1. UI 统一

- 替换核心页面中主要 emoji 语义
- 统一卡片、按钮、输入、标题层级
- 统一顶栏和导航风格

#### 2. 状态补齐

- 加载态
- 空状态
- 错误态
- 成功反馈

#### 3. 文案整理

- 首页文案
- 空状态文案
- 错误提示文案
- AI 说明文案

#### 4. 验收测试

- 新用户主链路测试
- 跨页联动测试
- AI 辅助测试

### 输出物

- 统一视觉的 MVP
- 验收记录

### 完成标志

- 用户可在稳定主链路下连续完成一天的基本工作管理

---

## 五、并行建议

### 可并行工作

1. 前端导航改造 与 后端 API 适配
2. 数据库迁移 与 UI 规范整理
3. 日历页实现 与 笔记页实现

### 不建议并行过深的工作

1. AI 助手与主页面结构同时大改
2. 视觉重构与页面结构同时大改

原因：

- 会造成重复返工
- 会把问题混在一起难以验收

---

## 六、风险点

### 风险 1：旧接口耦合过深

如果新页面继续大量依赖旧接口，MVP 契约会失效，页面行为也会继续漂移。

### 风险 2：主路径与低频功能抢资源

如果下载、工作流、同步继续抢排期，MVP 主链路很容易半成品。

### 风险 3：先做视觉后做结构

会导致今天页、任务页、日历页多次返工。

### 风险 4：AI 范围扩张

如果把 AI 定义成万能入口，会重新把复杂度带回系统。

---

## 七、实施完成标准

实施计划完成时，应满足：

1. MVP 页面结构稳定
2. MVP 接口稳定
3. 主链路能完整跑通
4. AI 只做辅助，不越界
5. 低频模块不影响核心体验

---

## 八、总结

这份实施计划的关键不是把任务排得很满，而是把顺序排对：

1. 先收口
2. 再打主链路
3. 再做联动
4. 最后统一视觉和验收

只要顺序错了，MVP 就会重新回到“大量模块并存，但没有一个主体验特别强”的旧状态。
