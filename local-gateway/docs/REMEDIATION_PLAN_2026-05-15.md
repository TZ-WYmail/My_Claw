# LocalCommandCenter 修复规划

> 文档版本: 1.0
> 更新日期: 2026-05-15
> 适用范围: `local-gateway/` 后端、前端、测试与文档

---

## 一、背景

本规划基于 2026-05-15 对仓库的静态审查与自动化测试结果整理，目标不是继续堆功能，而是先把当前系统从“可运行但高风险、契约漂移、测试不稳定”的状态，修复到“边界清晰、接口稳定、可验证”的状态。

本次审查结论可以概括为 4 类问题：

1. **安全边界失守**：未鉴权暴露 AI 对话、配置写入、本地代码执行能力
2. **隔离与执行模型不可靠**：所谓沙盒没有形成真实约束，超时和文件回传实现存在缺陷
3. **接口与构建产物漂移**：后端响应、前端 PWA 产物、测试断言、README 已出现多处分叉
4. **测试环境不确定**：运行结果依赖本地状态与外部 AI 服务，导致仓库不可稳定复现

---

## 二、当前问题总览

### 2.1 P0 - 必须优先修复

| 编号 | 问题 | 影响 | 说明 |
|------|------|------|------|
| P0-1 | `/api/chat` 未鉴权 | 极高 | 外部请求可驱动 AI 调用本地工具 |
| P0-2 | `/api/chat/config` 未鉴权且可持久化写配置 | 极高 | 可被匿名覆盖模型出口和 API Key |
| P0-3 | `code_interpreter` 在宿主机直接执行 | 极高 | 实际是宿主机代码执行，不是真沙盒 |
| P0-4 | `shell_exec` 在宿主机直接执行 | 极高 | 当前白名单不足以构成可信安全边界 |

### 2.2 P1 - 短期内必须收敛

| 编号 | 问题 | 影响 | 说明 |
|------|------|------|------|
| P1-1 | Docker 沙盒缺少真实超时与更严格资源限制 | 高 | 长任务和异常任务可能卡死或失控 |
| P1-2 | 沙盒输出文件复制实现错误 | 高 | 返回的产物可能是 tar 流而不是原文件 |
| P1-3 | `/api/search` 契约已变更但调用方未统一 | 高 | AI 工具、测试、前端可能使用不一致结构 |
| P1-4 | AI 规划接口绑定外部服务 | 高 | 测试和本地开发不稳定，离线不可复现 |
| P1-5 | AI 配置明文落盘 | 高 | 密钥管理风险较大 |

### 2.3 P2 - 中期清理

| 编号 | 问题 | 影响 | 说明 |
|------|------|------|------|
| P2-1 | PWA 产物命名与测试约定不一致 | 中 | `manifest.json` 缺失导致测试失败 |
| P2-2 | 标签服务非幂等 | 中 | 重复测试和重复调用容易报错 |
| P2-3 | 文档与实际实现偏差大 | 中 | 新开发和维护容易基于错误认知操作 |
| P2-4 | 根规范引用缺失文件 | 中 | 工作流规范本身失效 |

## 三、修复目标

### 3.1 安全目标

1. 所有高风险端点必须具备认证与授权
2. 宿主机不得直接暴露任意代码执行能力
3. AI 配置和密钥不得允许匿名修改
4. 沙盒执行必须具备清晰的资源、网络、文件边界

### 3.2 工程目标

1. 测试结果应在本地环境中稳定复现
2. API 契约只能保留一套主定义，兼容逻辑要显式
3. 前端构建产物、后端静态资源约定、测试断言必须统一
4. 文档描述必须回到“当前真实状态”，而非未来设想

---

## 四、分阶段修复方案

### Phase 0：立即止血

**目标**：先把最危险的能力收口，避免继续以不安全状态运行。

**任务清单**：

1. 为 `/api/chat`、`/api/chat/stream`、`/api/chat/config`、`/api/chat/test` 增加统一认证
2. 默认关闭 `code_interpreter` 和 `shell_exec`，改为显式配置开关
3. 在未完成重构前，禁止 AI 从 HTTP 路由间接触发宿主机执行
4. 为所有高风险调用增加审计日志，至少记录来源、时间、工具名、结果
5. 检查并轮换已经落盘或可能暴露的 AI Key

**验收标准**：

- 未携带合法凭证时，高风险端点返回 401/403
- 默认配置下，AI 无法执行宿主机代码或 shell
- 高风险操作可在日志中追踪

---

### Phase 1：重建执行边界

**目标**：把“本地执行能力”迁移到可控隔离层，而不是依赖字符串黑名单。

**任务清单**：

1. 将 `code_interpreter` 迁移到 Docker 沙盒执行链路，不再直接启动宿主机子进程
2. 评估是否保留 `shell_exec`；如果保留，只允许只读诊断命令并强制认证
3. 为 Docker 容器增加真实超时控制
4. 增加 CPU、PID、网络、只读根文件系统等限制
5. 收紧挂载白名单，移除对整个项目目录的默认暴露
6. 修复输出文件复制逻辑，正确解包 tar 流并校验目标文件

**验收标准**：

- 执行类功能不再直接依赖宿主机 `python3`、`node`、`/bin/bash`
- 超时任务可被确定性终止
- 输出文件可被正常打开且内容未损坏
- 沙盒无法访问非白名单宿主路径

---

### Phase 2：统一接口契约

**目标**：清理后端响应、AI 工具、前端调用和测试之间的分叉。

**任务清单**：

1. 明确 `/api/search` 的主契约：统一搜索还是文件搜索
2. 若 `/api/search` 保留统一搜索，则：
   - AI 工具改用 `/api/search/legacy` 或新增专用文件搜索端点
   - 前端、测试、schema 一并更新
3. 为所有关键端点补充响应结构测试
4. 审查 `models/schemas.py` 与真实返回值是否一致
5. 清理“旧接口兼容”但调用方仍走错入口的问题

**验收标准**：

- 同一功能只有一套明确主契约
- AI 工具返回结构与 schema 一致
- 搜索相关测试全部通过

---

### Phase 3：去除外部依赖导致的测试不稳定

**目标**：让仓库测试在无外网、无真实 AI Key 的环境下仍可稳定运行。

**任务清单**：

1. 为 AI 规划类接口增加可预测的本地 fallback
2. 测试环境强制禁用真实外部 AI 请求
3. 将 AI 能力测试改为 mock/fixture 驱动
4. 避免从本地持久化配置自动污染测试环境
5. 为测试提供独立数据库和独立运行目录

**验收标准**：

- `pytest` 在离线环境可稳定运行
- AI 规划测试不依赖真实供应商响应
- 测试不会读取用户本地残留配置造成分叉

---

### Phase 4：修复构建产物与前端集成

**目标**：统一前端源码、构建输出、后端静态目录和 PWA 约定。

**任务清单**：

1. 明确 PWA manifest 的标准文件名
2. 决定由 Vite 构建生成还是仓库内维护静态文件
3. 统一 `vite.config.js`、后端静态目录和测试断言
4. 检查 `base: '/static/'` 与根路由、图标路径、离线 fallback 是否一致
5. 清理前端 README 中的默认模板内容，改为项目真实说明

**验收标准**：

- `manifest.json` 或 `manifest.webmanifest` 的约定唯一且全链路一致
- PWA 相关测试全部通过
- 前端文档反映当前实现而非默认模板

---

### Phase 5：数据与文档治理

**目标**：降低运行态污染，收敛仓库事实来源。

**任务清单**：

1. 将 `tag_service.create_tag()` 改为幂等接口，或在重复名称时返回已存在结果
2. 检查所有 service 是否共享运行态数据库，补足测试隔离
3. 修正根 `AGENTS.md` 中失效引用
4. 修正根 README、子项目 README、架构文档中的过时描述
5. 标注哪些能力已经实现，哪些仍是规划项

**验收标准**：

- 重复运行测试不会因历史数据污染失败
- 仓库文档与实际代码目录一致
- 开发者能仅依赖仓库文档理解真实系统状态

---

## 五、建议实施顺序

建议不要并行修所有问题，优先按依赖关系推进：

1. **先做 Phase 0**：否则仓库仍持续暴露高风险能力
2. **再做 Phase 1**：否则“安全修复”仍只是黑名单补丁
3. **然后做 Phase 2 + Phase 3**：统一契约并恢复测试可复现性
4. **最后做 Phase 4 + Phase 5**：清理构建、文档和历史技术债

原因很直接：

- 如果不先关掉未鉴权执行面，其他修复都建立在高风险基础上
- 如果不先收口执行模型，之后的测试和接口统一仍会被“宿主机副作用”污染
- 如果不先统一 API 契约，前端和 AI 调用层会持续在错误接口上堆逻辑

---

## 六、里程碑定义

### M1 - 安全封网

完成条件：

- 高风险端点全部鉴权
- 宿主机直接执行默认关闭
- 密钥写入受控

### M2 - 执行收敛

完成条件：

- 所有执行能力走受控隔离层
- 沙盒超时、资源限制、文件回传可验证

### M3 - 契约稳定

完成条件：

- 搜索接口与 AI 工具契约统一
- PWA 产物与测试一致
- 测试可离线稳定通过

### M4 - 仓库可维护

完成条件：

- 文档与实现一致
- 测试与数据隔离
- 关键服务具备最小幂等性与边界说明

---

## 七、验证方案

每个阶段都需要对应验证，不建议只依赖手工点页面。

### 7.1 自动化验证

1. 补充认证失败与成功路径测试
2. 补充执行类能力的安全回归测试
3. 修复并稳定现有 `pytest` 套件
4. 为搜索契约和 PWA 产物增加固定断言

### 7.2 手工验证

1. 未授权访问高风险端点，确认被拒绝
2. 通过受控身份访问聊天能力，确认工具受限
3. 执行超时沙盒任务，确认容器被回收
4. 构建前端并验证根路径、图标、manifest、离线入口

---

## 八、风险与注意事项

1. **认证上线会影响现有客户端**：需要同步调整前端和任何外部调用方
2. **关闭宿主机执行会改变 AI 助手能力边界**：需要在产品层明确哪些能力保留、哪些必须改走 Docker
3. **统一搜索接口可能影响前端页面和 AI 工具调用**：必须一次性同步修改，避免双向兼容长期存在
4. **测试隔离改造会触及较多 service 层**：建议先抽出配置入口，再推进数据库与数据目录注入

---

## 九、结论

当前仓库不是“缺少几个小修补”，而是存在一组互相影响的系统性问题：安全边界、执行模型、接口契约和测试环境都没有完全收敛。

正确修复顺序应当是：

1. 先止血
2. 再收边界
3. 然后统一契约
4. 最后修复构建、文档和测试隔离

只有这样，后续新增功能才不会继续建立在不稳定和高风险的基础上。
