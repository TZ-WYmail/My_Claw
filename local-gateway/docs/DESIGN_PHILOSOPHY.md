# 全球贸易 AI 日程助手 — 开发设计思路文档

> **版本**: v1.0
> **日期**: 2026-04-28
> **定位**: 取竞品之长、补竞品之短、做竞品不做之事

## 零、设计总纲：我们要做什么

### 市场空白

经过对 15+ 款 AI 日程产品的深度调研，发现一个明确的市场空白：

**没有任何产品同时做到「多时区智能调度」+「AI 主动动态重塑日程」+「贸易业务深度绑定」+「本地优先隐私可控」。**

- **Motion / Reclaim**：AI 自动排程做得最好，但时区能力薄弱，且是纯 SaaS 云端产品，数据不在本地
- **Vimcal**：时区处理全球最强，但没有 AI 自动排程
- **SkedPal**：Time Map 精力分区概念最先进，但 UI 老旧、无移动端、无时区能力
- **飞书/钉钉**：中国生态最强，但 AI 日程只是附属功能，不做深度排程优化
- **Rise**：TimeFold 约束求解器技术最强，但已倒闭——教训是"不要让 AI 太激进地替用户做决定"

**我们的定位**：做一款 **本地部署的、以时区智能和动态重塑为双引擎的、贸易领域深度定制的 AI 日程系统**。

### 取长补短策略

| 竞品 | 我们要学的 | 我们要超越的 | 我们要避免的 |
|------|-----------|-------------|-------------|
| **Motion** | AI 全天候自动排程的思路、约束满足模型 | 它不做时区——我们做 | 太激进自动排程，用户失控感强 |
| **Reclaim.ai** | 习惯自动排入日程、缓冲时间保护 | 它的时区是基础的——我们做深度时区 | 依赖 Google Calendar，不能独立 |
| **SkedPal** | Time Map 精力分区、自然语言输入 | 它无移动端——我们做移动优先 | UI 过时、学习曲线陡 |
| **Vimcal** | 时区重叠可视化、极速键盘操作 | 它无 AI 排程——我们做 AI+时区 | 仅限会议排期，无任务管理 |
| **Rise** | TimeFold 约束求解器架构 | 它倒了的教训——用户不能失去控制 | Autopilot 太激进，自动挪会议被骂 |
| **飞书 My AI** | 中国生态集成思路、AI 助手统一入口 | 它不做深度日程优化——我们做 | 功能太多太杂，核心不突出 |
| **Amie** | 会议智能（录音→摘要→行动项） | 它不做排程——我们做全链路 | 仅聚焦会议，非完整日程系统 |
| **Trevor AI** | 亲民定价、人机混合（AI 建议 + 人确认） | 它不做时区——我们做 | AI 太弱，仅建议不行动 |

### 核心设计哲学

1. **用户主权原则**（Rise 教训）：AI 可以主动建议和自动执行低风险操作，但任何涉及已有日程变更的操作必须经用户确认。绝不悄悄挪走一个会议。
2. **时区原生原则**：不是在现有系统上"加时区支持"，而是从数据层到展示层全链路以 UTC+时区偏移为基本时间模型。
3. **领域深度原则**：不做通用工具的浅层叠加，而是深入贸易场景的任务链、合规日历、上下文关联。
4. **本地优先原则**：所有数据本地存储，AI 请求仅发送任务元数据，可选本地模型降级。

## 一、系统架构设计

### 1.1 整体架构：四层分离

```
┌─────────────────────────────────────────────────────┐
│                    展示层 (Presentation)              │
│   Web UI (SPA)  │  PWA Mobile  │  API (OpenAPI)     │
│   多时区渲染     │  离线优先     │  X-Timezone Header │
├─────────────────────────────────────────────────────┤
│                    调度层 (Scheduling)                │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ 时区引擎  │  │ 日程编排引擎  │  │  冲突消解引擎  │  │
│  │ Timezone │  │  Scheduler   │  │  ConflictRes  │  │
│  └──────────┘  └──────────────┘  └───────────────┘  │
├─────────────────────────────────────────────────────┤
│                    AI 层 (Intelligence)              │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ 优先级    │  │  精力模型    │  │  意图理解     │  │
│  │ Evaluator│  │ EnergyModel  │  │  IntentParser │  │
│  └──────────┘  └──────────────┘  └───────────────┘  │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ 日程生成  │  │  主动建议    │  │  任务拆解     │  │
│  │ DayPlanner│  │  Proactive   │  │  Decomposer   │  │
│  └──────────┘  └──────────────┘  └───────────────┘  │
├─────────────────────────────────────────────────────┤
│                    数据层 (Data)                      │
│  SQLite (UTC)  │  文件存储  │  同步引擎  │  备份     │
│  时区元数据     │  关联索引   │  冲突合并  │  增量     │
└─────────────────────────────────────────────────────┘
```

**设计思路**：

- **调度层与 AI 层分离**：调度层是确定性的算法（约束求解、时区计算），AI 层是概率性的推理（优先级评估、精力建模）。两层协作但不耦合——即使 AI 层不可用，调度层仍能基于规则完成基本排程。这解决了 Rise 的教训：当 AI 做出错误判断时，确定性调度层可以作为安全网兜底。
- **展示层不关心时区计算**：展示层只负责"拿到 UTC 时间 + 目标时区 → 渲染显示"，所有时区逻辑下沉到调度层。API 层通过 `X-Timezone` header 让客户端声明当前时区，服务端统一转换。

### 1.2 数据模型：UTC 为王

**现状问题**：当前 `tasks` 表的 `due_time` 是 `TEXT` 类型，存储的是无时区信息的本地时间字符串。`calendar_events` 同理。这导致跨时区时无法正确转换。

**设计思路**：

所有时间字段统一采用 **ISO 8601 with timezone** 格式存储：

```
存储格式: "2026-04-28T09:00:00+08:00"
内部计算: 转为 UTC timestamp
显示输出: 根据请求时区转换
```

**改造要点**：

- `due_time`、`start_time`、`end_time` 等字段保留 TEXT 类型（SQLite 无原生时区支持），但强制要求写入时带时区偏移
- 新增 `timezone_meta` 字段（JSON），记录"创建时时区""关联方时区"等元信息，用于智能提示
- 所有时间比较和排序在服务层转为 UTC 进行，避免时区陷阱
- 迁移策略：对旧数据，用创建时的默认时区（从 `config.py` 读取或用户设置）补全偏移量

### 1.3 模块交互流程

以"晨间日程生成"为例，展示四层协作：

```
1. 触发: 定时任务(07:00) 或 用户打开首页
      ↓
2. 调度层·时区引擎:
   - 获取用户当前时区(UTC+8)
   - 检测最近7天时区变化(从UTC-5切换)
   - 计算时差恢复系数(0.6)
      ↓
3. 数据层:
   - 查询今日所有待办(due_time在今日范围内,UTC)
   - 查询今日日历事件
   - 查询未完成任务(按优先级排序)
      ↓
4. AI层:
   a. 优先级评估器:
      - 对每个任务计算"动态优先级分数" = 基础优先级 × 时效衰减 × 业务权重
      - 标记"不可移动"任务(截止时间固定)
   b. 精力模型:
      - 根据用户精力曲线 + 时差恢复系数
      - 为每个时段标注"认知负荷容量"
   c. 日程编排器:
      - 将任务分配到匹配的时段
      - 高认知任务→高精力时段
      - 插入缓冲时间
      - 检测冲突
      ↓
5. 调度层·冲突消解:
   - 检测时间重叠、精力冲突、依赖冲突
   - 生成解决方案(自动/建议)
      ↓
6. 展示层:
   - 渲染日程表(30分钟粒度)
   - 时区标注(本地+原始+关联方)
   - 精力曲线背景色
   - 冲突提示+解决方案按钮
```

## 二、时区智能引擎设计

### 2.1 设计目标

对标 Vimcal 的时区重叠可视化 + 超越它的 AI 时区推理能力。

### 2.2 核心数据结构

```
用户时区档案 (UserTimezoneProfile):
  - home_timezone: "Asia/Shanghai"       // 常驻地
  - current_timezone: "Asia/Dubai"       // 当前所在地(自动检测/手动)
  - timezone_history: [                   // 近期时区变化轨迹
      {tz: "America/New_York", from: "2026-04-25", to: "2026-04-27"},
      {tz: "Asia/Shanghai", from: "2026-04-27", to: "2026-04-28"}
    ]
  - work_hours: {                         // 工作时间偏好(本地时间)
      start: "09:00", end: "18:00",
      lunch: {"12:00-13:00"},
      no_meeting_hours: []                // 禁止会议时段
    }
  - jet_lag_sensitivity: "medium"         // 时差敏感度: low/medium/high
```

### 2.3 时区检测机制

设计三级检测，从自动到手动：

1. **自动检测（优先）**：通过浏览器 `Intl.DateTimeFormat().resolvedOptions().timeZone` 获取客户端时区，通过 API `X-Timezone` header 传递给服务端。移动端通过 GPS 坐标 + IANA Time Zone Boundary Map 推算。
2. **半自动推断**：当自动检测不可用时，根据用户的日历事件时区分布推断当前所在时区。例如，连续 3 天的事件都标注为 "Asia/Dubai"，则推断用户在迪拜。
3. **手动设置**：用户可在设置中手动切换当前时区。提供常用时区快捷列表（贸易商常用：上海、迪拜、伦敦、纽约、新加坡、东京）。

**时区变化检测**：当检测到用户时区与上次记录不同时，触发以下动作：
- 记录时区变化到 `timezone_history`
- 计算时差幅度（小时差）
- 如果时差 ≥ 3 小时，触发时差恢复模式
- 重新评估今日日程（异步，不阻塞用户操作）

### 2.4 跨时区会议窗口算法

这是 Vimcal 的核心能力，我们要实现并增强：

**Step 1：收集参与者时间窗口**

```
参与者 A: Asia/Shanghai (UTC+8), 工作时间 09:00-18:00 本地
参与者 B: Asia/Dubai (UTC+4), 工作时间 09:00-17:00 本地
参与者 C: Europe/London (UTC+1), 工作时间 09:00-17:00 本地
```

**Step 2：转换为 UTC 基准**

```
A: 01:00-10:00 UTC
B: 05:00-13:00 UTC
C: 08:00-16:00 UTC
```

**Step 3：计算交集**

```
交集: 08:00-10:00 UTC (2小时窗口)
```

**Step 4：舒适度评分**（这是我们的增强点，Vimcal 不做）

每个潜在时段计算一个舒适度分数，维度包括：

| 维度 | 权重 | 说明 |
|------|------|------|
| 时段位置 | 30% | 在各方工作日的哪个位置？（上午精力好 > 下午 > 边缘时间）|
| 用餐回避 | 20% | 是否踩到任一方的午餐时间（12:00-13:00 本地）|
| 精力匹配 | 20% | 是否在各方的高精力时段（09:00-11:00 本地）|
| 非工作边缘 | 15% | 是否在某人工作时间的开始/结束 30 分钟内 |
| 会议密度 | 15% | 该时段前后是否已有会议（避免连续开会）|

**Step 5：推荐排序**

输出 Top 3 时段，按舒适度降序，每个时段标注：
- 各方的本地时间
- 舒适度分数和关键因素
- "建议"文案（如"此时段对伦敦方最优，上海方在下班前但可接受"）

### 2.5 时差恢复模型

**设计思路**：受 SkedPal 的 Time Map 概念启发，但不是让用户手动定义时间分区，而是 AI 根据时差自动生成"恢复期 Time Map"。

**恢复系数计算**：

```
时差幅度 Δh = |新时区UTC偏移 - 旧时区UTC偏移|
恢复天数 D = ceil(Δh / 2)  (经验公式: 每跨越2个时区约需1天恢复)
恢复系数 R(t) = 1 - (1 - t/D) × 0.4  (Day1最低0.6, 线性恢复)

其中 t = 距离时区切换的天数 (0-indexed)
R=1.0 表示完全恢复, R=0.6 表示建议只安排60%负荷
```

**恢复模式对日程的影响**：

| 恢复系数 | 日程密度上限 | 单次专注上限 | 避免安排 |
|----------|-------------|-------------|---------|
| ≤0.5 | 4h/天 | 45min | 重要谈判、高认知决策 |
| 0.5-0.7 | 5-6h/天 | 60min | 连续3h以上的深度工作 |
| 0.7-0.9 | 6-7h/天 | 90min | 高强度多任务切换 |
| ≥0.9 | 正常 | 正常 | 无限制 |

用户可手动关闭恢复模式（`jet_lag_sensitivity: "low"` 的用户自动跳过）。

## 三、AI 动态日程引擎设计

### 3.1 设计目标

对标 Motion 的全自动排程 + Reclaim 的习惯/缓冲保护 + SkedPal 的 Time Map，但要避免 Rise 的 Autopilot 教训。

### 3.2 优先级动态评估模型

**现状**：当前系统的 `priority` 是静态整数（0=urgent, 1=high, 2=medium, 3=low），创建后不变。

**设计思路**：引入"动态优先级分数"（Dynamic Priority Score, DPS），每个任务在每次评估时重新计算。

```
DPS = w1 × 基础优先级 + w2 × 时效紧迫度 + w3 × 业务权重 + w4 × 依赖阻塞度

其中:
- 基础优先级: 用户设定的 priority (0-3) 归一化到 0-1
- 时效紧迫度: f(距截止时间) 越近越高，指数衰减函数
  urgency(t) = e^(-α × (deadline - now) / total_window)
  α=3 时，截止前24h内紧迫度急剧上升
- 业务权重: 贸易场景专用权重系数
  - 报关/船期/信用证截止: ×1.5
  - 客户会议/谈判: ×1.3
  - 供应商跟进: ×1.1
  - 内部事务/行政: ×1.0
- 依赖阻塞度: 如果此任务是后续任务的阻塞项，乘以阻塞因子
  blocked_factor = 1 + 0.3 × (被阻塞任务数)
```

**评估时机**：
- 每日首次打开（晨间日程生成）
- 新任务创建时
- 任务截止时间变更时
- 外部事件到达时（Webhook 触发）
- 每 30 分钟定时微调（低优先级后台任务，不打扰用户）

### 3.3 精力感知排程

**设计思路**：融合 SkedPal 的 Time Map 概念和 Rise 的精力保护理念，但用 AI 自适应而非用户手动定义。

**默认精力曲线模型**（基于认知科学研究）：

  08:00  ████████████████░░░░  80  (进入高峰)
  09:00  ██████████████████░░  90  (黄金时段)
  10:00  ████████████████████  100 (峰值)
  11:00  ██████████████████░░  90  (仍处高峰)
  12:00  ██████████░░░░░░░░░░  50  (午餐低谷)
  13:00  ████████████░░░░░░░░  60  (午休恢复)

**任务认知负荷分类**：

| 负荷等级 | 典型任务 | 所需精力 | 建议时段 |
|---------|---------|---------|---------|
| 极高 | 重要谈判、复杂报价、合同审核 | ≥85 | 09:00-11:00 |
| 高 | 客户沟通、决策型会议、方案设计 | ≥70 | 09:00-11:00, 14:00-15:00 |
| 中 | 常规跟进、文档编写、团队沟通 | ≥50 | 任何工作时间 |
| 低 | 邮件回复、数据整理、行政事务 | ≥30 | 12:00-13:00, 16:00-18:00 |

**个性化**：AI 根据用户的番茄钟数据、任务完成时间分布，自动校准精力曲线。例如，如果用户连续 30 天在 20:00-22:00 完成高认知任务，AI 会将晚间标记为可选的高精力时段。

### 3.4 日程编排算法

**核心思路**：约束满足问题（CSP）+ 贪心启发式。

**Motion 的做法**：将日程视为 CSP，用约束求解器找到最优解。但这导致每次变更都要重算整个日程，计算量大且用户感知变化太剧烈。

**我们的做法**：分层编排 + 增量调整。

```
┌─────────────────────────────────────────┐
│  Layer 1: 固定锚点（不可移动）            │
│  - 截止时间固定的任务                     │
│  - 已确认的会议                           │
│  - 用户手动锁定的时段                     │
│  → 这些是约束，不被算法移动               │
└─────────────────────────────────────────┘
          ↓ 剩余时段
┌─────────────────────────────────────────┐
│  Layer 2: 高优先级弹性任务                │
│  - DPS 排名 Top 30% 的任务               │
│  - 分配到最佳精力匹配时段                  │
│  → 算法自动排入，用户可调整               │
└─────────────────────────────────────────┘
          ↓ 剩余时段
┌─────────────────────────────────────────┐
│  Layer 3: 低优先级填充任务                │
│  - DPS 排名后 70% 的任务                  │
│  - 填入剩余空档                           │
│  → 算法建议，用户确认                     │
└─────────────────────────────────────────┘
          ↓ 间隙
┌─────────────────────────────────────────┐
│  Layer 4: 缓冲与保护                     │
│  - 会议间 15 分钟缓冲（Reclaim 理念）     │
│  - 习惯时间块（Reclaim 理念）             │
│  - 深度工作时间块（Clockwise 理念）        │
│  → 算法自动保护，用户可覆盖               │
└─────────────────────────────────────────┘
```

**增量调整规则**（避免 Rise 的"整日重塑"问题）：

- 当新任务到达时，仅尝试插入空档；空档不足时，仅移动 Layer 3 的低优先级任务
- 当任务完成时，释放的时段不立即重排，而是标记为"弹性时间"供用户自由支配
- 当外部事件打断时（如会议超时），仅调整当前时段之后的任务，不回溯修改已完成的安排
- 每次调整最多移动 3 个任务（防止变化过于剧烈）

### 3.5 用户控制与 AI 自主的平衡（Rise 教训的具体应对）

Rise 的 Autopilot 之所以失败，核心原因是用户感到失控。我们的设计原则：

| 操作类型 | AI 权限 | 用户参与 | 通知级别 |
|---------|--------|---------|---------|
| 创建新任务 | 全自动执行 | 无需确认 | 静默（任务列表可见）|
| 标记任务完成 | 全自动执行 | 无需确认 | 静默 |
| 插入空档排程 | 自动执行 | 无需确认 | 日程变更摘要 |
| 移动低优先级任务 | 建议模式 | 需确认 | 推送建议 |
| 移动高优先级任务 | 禁止自动 | 用户手动 | — |
| 改期已有会议 | 禁止自动 | 用户手动 + 发送改期请求 | — |
| 时差恢复降负荷 | 建议模式 | 需确认 | 晨间日程中标注 |
| 紧急事件重塑 | 建议模式 | 需确认 | 强推送 |

**设计细节**：所有 AI 建议都有一个"信任度分数"。随着用户不断采纳 AI 建议，信任度提升，部分操作可以从"建议模式"升级为"自动执行"。用户也可以手动设置信任等级（保守/均衡/信任）。

### 3.6 晨间日程生成详细流程

这是核心用户体验，详细设计流程：

```
07:00 触发
  │
  ├─ Step 1: 环境感知 (5秒内)
  │   ├─ 获取当前时区
  │   ├─ 检测时差恢复状态
  │   ├─ 获取今日天气（可选，影响外出安排）
  │   └─ 扫描昨夜到达的新消息/邮件（通过Webhook或手动标记）
  │
  ├─ Step 2: 任务盘点 (3秒内)
  │   ├─ 查询今日截止任务
  │   ├─ 查询今日日历事件
  │   ├─ 查询逾期未完成任务
  │   ├─ 查询即将到期任务（未来3天）
  │   └─ 计算每个任务的 DPS
  │
  ├─ Step 3: 日程编排 (5秒内)
  │   ├─ 固定锚点：截止时间不可变的任务 → 锁定时段
  │   ├─ 精力匹配：高DPS任务 → 分配到高精力时段
  │   ├─ 缓冲插入：会议前后 → 15分钟缓冲
  │   ├─ 习惯保护：用户的习惯时间块 → 保留
  │   ├─ 弹性填充：低DPS任务 → 填入空档
  │   └─ 冲突检测 → 生成解决方案
  │
  ├─ Step 4: 生成摘要
  │   ├─ 今日概览："今日8项待办，2项紧急，1项截止"
  │   ├─ 风险提示："报关截止15:00，但资料未齐"
  │   ├─ 时差提醒："您刚从纽约回来，建议负荷降至60%"
  │   └─ 智能建议："15:00的内部会议可延至明日，为报关留足时间"
  │
  └─ Step 5: 推送
      ├─ 移动端推送通知（摘要）
      ├─ Web UI 首页展示（完整日程）
      └─ 可选：邮件/微信推送
```

**性能要求**：整个流程 ≤ 10 秒（不含用户确认等待）。

## 四、贸易业务层设计

### 4.1 任务链模板系统

**现状**：当前系统有 `subtasks` 表支持子任务拆分，有 `recurrence` 支持重复周期。但不支持"任务链"——一系列有依赖关系的任务的模板化复用。

**设计思路**：

```
任务链模板 (TaskChainTemplate):
  id: "tc_fob_export"
  name: "FOB出口标准流程"
  description: "从订单确认到尾款回收的完整流程"
  owner_timezone: "Asia/Shanghai"
  nodes: [
    {
      id: "n1",
      name: "确认订单细节",
      offset_days: -30,          // 相对于基准日(船期)的偏移天数
      priority: "high",
      cognitive_load: "high",
      business_weight: 1.5,
      estimated_minutes: 120,
      depends_on: []             // 无前置依赖
    },
    {
      id: "n2",
      name: "备货完成确认",
      offset_days: -21,
      priority: "high",
      cognitive_load: "medium",
      business_weight: 1.3,
      estimated_minutes: 60,
      depends_on: ["n1"]
    },
    // ... 更多节点
  ]

实例化 (TaskChainInstance):
  chain_id: "tc_fob_export"
  base_date: "2026-05-28"       // 船期(基准日)
  context: {
    order_no: "PO-2026-0415",
    supplier: "鑫达塑胶",
    customer: "Al Fahim Trading",
    currency: "USD",
    amount: 25000
  }
  → 自动生成8个子任务,每个带计算好的due_time
```

**级联更新机制**：

当任务链中某个节点延期时：

```
延期节点: "备货完成确认" 延期3天
  │
  ├─ 检查下游依赖: "订舱确认" → depends_on: ["备货完成确认"]
  │   └─ 延期3天
  │       ├─ 检查下游: "报关准备" → also 延期3天
  │       │   └─ 检查下游: "装柜" → also 延期3天
  │       │       └─ 检查: 是否影响基准日(船期)?
  │       │           └─ 如果船期不可变: 从"装柜"到"截单"时间被压缩
  │       │               → 触发告警: "装柜到截单时间不足,需加急处理"
  │       └─ 如果船期可变: 通知物流方改期
  │
  └─ 生成影响报告:
     "备货延期3天,影响4项下游任务:
      - 订舱确认: 5月7日→5月10日
      - 报关准备: 5月14日→5月17日
      - 装柜安排: 5月25日→5月28日
      - 截单: 5月28日(不可变)→ 装柜到截单仅0天!
      ⚠️ 建议联系货代确认是否可推迟截单"
```

### 4.2 实体关联模型

**现状**：当前系统有标签（`tags`/`task_tags` 表），但标签是扁平的，无法表达"这个任务属于哪个订单""这个文件关联哪个供应商"。

**设计思路**：引入轻量级实体关联层，不做完整知识图谱（太重），而是用"关联组"概念。

```
关联组 (ContextGroup):
  group_id: "cg_PO-2026-0415"
  group_type: "order"           // order/supplier/customer/shipment
  primary_key: "PO-2026-0415"
  display_name: "PO-2026-0415 迪拜Al Fahim订单"
  metadata: {
    supplier: "鑫达塑胶",
    customer: "Al Fahim Trading",
    amount: "USD 25,000",
    incoterm: "FOB Shanghai",
    status: "in_progress"
  }

关联 (ContextLink):
  - task_id → group_id
  - note_id → group_id
  - calendar_event_id → group_id
  - download_id → group_id (关联的合同/报关单文件)
```

**AI 上下文查询**：用户问"PO-2026-0415 进度如何"，AI 自动：
1. 查找关联组 `cg_PO-2026-0415`
2. 汇总关联任务状态（5个任务，2完成2进行1逾期）
3. 汇总关联文件（合同已下载，报关单待准备）
4. 汇总近期日历事件（明日与供应商的视频会议）
5. 生成摘要："该订单8个节点中已完成2个，当前卡在备货确认（逾期2天），明日有供应商会议可推进"

### 4.3 合规日历

**设计思路**：不是做一个完整的合规数据库，而是提供一个"合规事件源"框架，通过 Webhook/数据源接入，AI 根据任务内容自动关联合规风险。

```
合规事件源 (ComplianceEventSource):
  - 中国海关公告 (通过RSS/Webhook)
  - 各国节假日 (内置IANA数据)
  - 用户自定义的资质到期日
  - 行业预警（用户订阅的Webhook）

合规关联规则:
  - 如果任务涉及"出口到X国" 且 X国有即将到来的政策变更 → 告警
  - 如果任务链的截单日落在目的国节假日 → 提前告警"目的港节假日，可能影响清关"
  - 如果信用证到期日落在银行休假日 → 提前告警"到期日为银行假日，须提前交单"
```

## 五、通知与预警系统设计

### 5.1 设计目标

对标 Reclaim 的缓冲时间保护 + 超越它的分级预警和场景化通知。

### 5.2 通知分级框架

**设计思路**：借鉴医疗系统的分诊（Triage）概念，而非简单的 urgent/normal。

```
通知分级 (NotificationTier):
  ╔═══════════════════════════════════════════════╗
  ║  T1 · 熔断 (Circuit Breaker)                  ║
  ║  船期截单<1h、信用证到期<24h、报关即将截止     ║
  ║  → 强推送+声音+震动+屏幕常亮+需手动确认       ║
  ╠═══════════════════════════════════════════════╣
  ║  T2 · 紧急 (Urgent)                           ║
  ║  高DPS任务即将到期(24h)、会议变更、             ║
  ║  任务链级联延期影响基准日                       ║
  ║  → 标准推送+角标+下次打开时首页高亮             ║
  ╠═══════════════════════════════════════════════╣
  ║  T3 · 建议 (Advisory)                         ║
  ║  AI日程优化建议、时差恢复提示、                  ║
  ║  习惯打卡提醒、日程冲突建议                     ║
  ║  → 静默推送/角标/下次打开时展示                ║
  ╠═══════════════════════════════════════════════╣
  ║  T4 · 信息 (Informational)                    ║
  ║  每日日程摘要、完成统计、物流状态更新            ║
  ║  → 仅在用户主动查看时展示                      ║
  ╚═══════════════════════════════════════════════╝
```

### 5.3 智能提醒时机

**设计思路**：不是简单的"提前N分钟提醒"，而是根据任务类型和用户场景动态调整。

```
提醒策略矩阵:

任务类型         │ 提醒序列
────────────────┼──────────────────────────────────
报关截止         │ T-24h(T3) → T-6h(T2) → T-2h(T1) → T-30min(T1)
船期截单         │ T-24h(T3) → T-12h(T2) → T-6h(T2) → T-1h(T1)
信用证到期       │ T-7d(T3) → T-3d(T2) → T-24h(T1)
客户会议         │ T-15min(T2) [+ 跨时区额外T-5min时区确认]
供应商跟进       │ T-30min(T3)
内部常规任务     │ T-15min(T3)
习惯打卡         │ 偏好时间(T4)，逾期2h后(T3)

特殊场景叠加:
- 时差恢复期: 所有T3/T4降级为T4（不推送），T1/T2保持
- 免打扰时段: T1突破免打扰（震动），T2-T4静默到时段结束
- 连续提醒抑制: 同一任务24h内最多4次提醒
```

### 5.4 预警触发器

**设计思路**：在现有 `workflows` 的触发器-动作模型上，增加"条件监控"型触发器。

```
条件监控触发器 (ConditionMonitor):
  monitor_id: "mon_fx_usdcny"
  condition: "USD/CNY 汇率变动 > 0.5% (24h内)"
  data_source: "webhook:fx_alert"
  actions:
    - type: "create_task"
      name: "评估锁汇策略"
      priority: "high"
      cognitive_load: "high"
    - type: "notify"
      tier: "T2"
      message: "USD/CNY 变动0.7%，影响3笔未结订单，预计损益 ¥12,500"
    - type: "insert_schedule"
      duration: 30  # 分钟
      label: "汇率分析时间"
      prefer_time: "today_afternoon"
```

## 六、AI 交互设计

### 6.1 设计目标

对标 Amie 的会议智能 + Trevor AI 的对话式交互 + 飞书 My AI 的统一入口，但聚焦在日程规划领域。

### 6.2 意图理解层

**设计思路**：在现有 `ai_service.py` 的 function calling 框架上，增加"意图预处理层"，让 AI 更好地理解贸易场景语言。

```
意图预处理流程:
  用户输入: "下周一前把迪拜那个单子的信用证改了"
      ↓
  实体识别:
    - 时间: 下周一 → 2026-05-04
    - 地点: 迪拜 → 关联 Al Fahim 订单
    - 业务: 信用证修改 → L/C Amendment
    - 动作: 修改/确认
      ↓
  上下文解析:
    - "那个单子" → 查找最近关联"迪拜"的活跃订单 → PO-2026-0415
    - "信用证改了" → 匹配任务链模板中的"信用证修改"节点
      ↓
  AI 理解结果:
    {
      intent: "create_task",
      task_name: "确认信用证修改 - PO-2026-0415",
      due_time: "2026-05-04T18:00:00+08:00",
      priority: 1,  // 信用证相关=高优先级
      business_weight: 1.5,
      context_group: "cg_PO-2026-0415",
      cognitive_load: "high"
    }
```

**技术路线**：不需要训练专用模型。利用现有 LLM（GLM-4 / GPT-4o）的 function calling 能力，设计更精细的工具 schema，让 AI 在调用工具前先做实体识别和上下文关联。关键是在 `TOOLS_SCHEMA` 中增加贸易领域参数和上下文查询工具。

### 6.3 主动建议引擎

**设计思路**：借鉴 Reclaim 的"AI Hours"和飞书 My AI 的主动推荐，但做得更场景化。

```
主动建议触发条件:

1. 时间触发:
   - 每日 07:00 → 晨间日程生成
   - 每日 17:00 → 明日预览 + 今日总结
   - 每周五 16:00 → 周报 + 下周预览

2. 事件触发:
   - 任务逾期 → "这个任务已逾期，需要调整截止时间或优先处理？"
   - 任务链节点延期 → 级联影响分析 + 建议
   - 日程冲突检测 → 冲突解决方案
   - 新消息/Webhook到达 → 提取待办 + 建议排入日程

3. 模式触发:
   - 连续3天未处理某高优先级任务 → "这批货的报关资料准备进度如何？"
   - 今日日程>8h → "今日任务较重，建议将以下2项移至明日"
   - 本周会议>10场 → "本周会议密度偏高，建议合并或取消部分"
   - 时差恢复期检测 → 自动降负荷建议

4. 洞察触发:
   - 每月1日 → 上月效率报告 + 本月建议
   - 检测到用户长期在非工作时间处理高认知任务 → 建议调整精力曲线
   - 检测到某类任务频繁延期 → 分析原因并建议
```

**建议质量保障**：每条主动建议必须包含：
1. **事实依据**：为什么给这个建议（数据支撑）
2. **行动方案**：具体要做什么（可执行）
3. **可选范围**：不只是"是/否"，提供"稍后/自定义/忽略"选项
4. **学习反馈**：用户忽略此建议后，类似场景降低建议频率

### 6.4 对话式日程操作

**设计思路**：当前 AI 通过 function calling 操控工具，但用户无法自然地"对话式"调整日程。增强方向：

```
用户: "今天下午太满了，能不能把供应商会议挪到明天？"
AI:
  1. 解析意图: 移动事件 + 时间偏好("明天") + 事件识别("供应商会议")
  2. 查找匹配: 今日14:00 "鑫达塑胶验厂视频" (匹配"供应商会议")
  3. 查询明日可用时段: 10:00-12:00, 14:00-17:00
  4. 生成方案:
     "找到1个匹配：14:00的'鑫达塑胶验厂视频'
      明日推荐时段：10:00-10:30 或 14:30-15:00
      注意：对方在东莞(UTC+8)，时段无时差问题
      确认移动到明日10:00？"
  5. 用户确认 → 执行移动 + 释放今日14:00时段 + 重新填充空档
```

## 七、移动端与离线设计

### 7.1 设计目标

飞书级移动体验 + 超越它的离线能力。

### 7.2 移动端架构

```
移动端技术选型:
  ┌──────────────────────────────────────┐
  │           PWA (Progressive Web App)  │
  │   - 离线缓存 (Service Worker)        │
  │   - 推送通知 (Web Push API)          │
  │   - 主屏幕安装                        │
  │   - 无需应用商店分发                   │
  └──────────────────────────────────────┘

  备选方案（长期）:
  ┌──────────────────────────────────────┐
  │     React Native / Flutter 原生APP   │
  │   - 更好的语音交互体验                │
  │   - 更好的GPS/时区检测                │
  │   - 更好的离线AI（本地模型集成）       │
  └──────────────────────────────────────┘
```

**选择 PWA 作为 MVP 理由**：
- 当前已有完整 Web UI，PWA 改造成本最低
- 贸易商人手机上已有微信/WhatsApp/邮件等占内存的 APP，PWA 无需安装
- Service Worker 可实现离线缓存核心页面
- Web Push API 可实现推送通知（Chrome/Edge 支持，Safari 16+ 支持）

### 7.3 离线模式设计

```
离线能力分层:

  Level 1 — 完全离线可用:
    - 查看今日日程 (Service Worker缓存)
    - 创建/完成任务 (IndexedDB本地暂存)
    - 语音备忘录 (本地录音+Whisper识别)

  Level 2 — 降级但可用:
    - 语音创建任务 (离线Whisper识别 + 本地简单解析,
                    联网后AI补全语义)
    - 日程查看 (缓存最近7天, 超出范围不可用)
    - 任务搜索 (仅本地IndexedDB数据)

  Level 3 — 不可用:
    - AI对话 (需要远程API)
    - 全文搜索 (需要服务端索引)
    - 跨设备同步 (需要网络)
    - 沙盒执行 (需要Docker)
```

**数据同步策略**：

```
同步协议 (基于现有sync_service扩展):
  1. 客户端记录本地操作日志 (OperationLog)
  2. 联网后推送操作日志到服务端
  3. 服务端按时间戳顺序重放操作
  4. 冲突检测: 同一任务被两端修改
     → 最后修改优先 (Last-Write-Win)
     → 如果字段级冲突: 提示用户选择
  5. 同步完成后, 客户端拉取最新全量数据
```

## 八、安全与隐私设计

### 8.1 数据安全

```
安全层次:

  传输层:
    - 远程访问时强制 TLS 1.3 (通过ngrok/Caddy/Nginx反向代理)
    - API 认证: Bearer Token (JWT) + API Key 双模式
    - WebSocket (推送): 同源策略 + Token 验证

  存储层:
    - SQLite 数据库: 本地文件权限 600
    - 可选: SQLCipher 加密数据库 (性能损耗约15%)
    - 敏感字段 (API Key等): AES-256 加密存储

  AI 请求层:
    - 本地模型优先: Ollama + Qwen-7B 做基础推理
    - 远程API降级: 仅发送任务元数据(名称/优先级/截止时间/标签)
    - 脱敏规则: 不发送金额、客户名称、订单号等业务敏感信息
    - 用户可配置: 选择"仅本地模型"或"允许远程API"
```

### 8.2 访问控制

```
角色模型:
  Owner (所有者):
    - 完全控制
    - 管理其他角色权限

  Assistant (助理):
    - 查看日程、任务
    - 创建/修改任务
    - 发送会议邀约
    - 不可删除任务/不可修改配置

  Viewer (只读):
    - 查看日程、任务
    - 查看笔记/文件
    - 不可任何修改操作

认证方式:
  - 本地使用: 无需认证 (localhost)
  - 远程使用: API Key (简单场景) / JWT (角色场景)
  - 可选 TOTP 双因素
```

### 9.1 新增依赖

## 十、竞品差异化总结

```
                    Motion  Reclaim  SkedPal  Vimcal  飞书   我们
                    ─────  ───────  ───────  ──────  ────   ───
AI 全天排程          ✅      ✅       ✅       ❌      ⚠️    ✅
多时区智能           ❌      ❌       ❌       ✅      ❌    ✅✅
时差恢复模式         ❌      ❌       ❌       ❌      ❌    ✅
精力感知排程         ⚠️      ❌       ✅       ❌      ❌    ✅
用户可控 + AI 自主   ❌      ✅       ⚠️       ❌      ❌    ✅
贸易任务链           ❌      ❌       ❌       ❌      ❌    ✅
实体上下文关联       ❌      ❌       ❌       ❌      ⚠️    ✅
分级通知+智能时机    ❌      ⚠️       ❌       ❌      ❌    ✅
本地部署/隐私可控    ❌      ❌       ❌       ❌      ❌    ✅
离线可用             ❌      ❌       ❌       ❌      ⚠️    ✅
中国生态集成         ❌      ❌       ❌       ❌      ✅    ✅
会议智能             ❌      ❌       ❌       ❌      ✅    ⚠️
```

**我们的三大差异化壁垒**：
1. **时区原生 + AI 排程的深度融合**：竞品要么做好时区（Vimcal）要么做好排程（Motion/Reclaim），没有产品同时做好两者并让 AI 在时区约束下做智能决策
2. **贸易领域深度绑定**：任务链模板 + 级联更新 + 实体关联 + 合规日历，这是通用工具不会做的垂直深度
3. **本地优先 + 用户主权**：数据本地存储、AI 行为可控可审计、离线可用——这是所有 SaaS 竞品无法提供的
